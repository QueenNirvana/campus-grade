# 高校成绩管理系统后端技术流程与代码讲解

> 重点：这份文档主要讲后端 Java 技术流程。前端只作为“请求从哪里来”的辅助说明。  
> 目标：能把本项目从启动、登录、鉴权、权限、业务处理、MyBatis 查库、成绩计算、统计返回这一整条链路讲清楚。

## 0. 先讲清楚：这个后端项目到底怎么运转

这个后端本质上是一个运行在 JVM 里的 Java Web 服务。它启动以后会监听 `8080` 端口，等待前端发来的 HTTP 请求。前端只是把请求发过来，真正决定“谁能访问、查什么数据、怎么计算成绩、怎么保存数据库”的都是后端。

可以把整个后端理解成一条流水线：

```text
HTTP 请求进入 8080 端口
  -> 内置 Tomcat 接收请求
  -> Spring Security 过滤器链先检查安全问题
  -> JWT 过滤器解析 token，确认当前用户
  -> Spring MVC 根据 URL 找到 Controller 方法
  -> Controller 把 JSON 参数转成 Java DTO
  -> Controller 调用 Service
  -> Service 执行业务规则、权限校验、成绩计算
  -> Service 调用 Mapper 接口
  -> MyBatis 根据 Mapper 接口找到 XML SQL
  -> JDBC 驱动把 SQL 发给 MySQL
  -> MySQL 返回结果
  -> MyBatis 把结果封装成 Java 对象
  -> Service 返回业务结果
  -> Controller 包装成统一 JSON
  -> 前端收到结果并显示
```

这条链路就是这个项目后端的核心运转流程。

### 0.1 后端为什么能接收网页请求

因为 Spring Boot Web 项目里内置了 Tomcat。

传统 Java Web 项目通常要把项目打成 war 包，放到外部 Tomcat 里运行。Spring Boot 更简单，它把 Tomcat 内嵌进项目里。执行：

```powershell
mvn spring-boot:run
```

后，项目会启动一个内置 Tomcat，并监听：

```text
localhost:8080
```

所以当前端请求：

```text
http://localhost:8080/api/grades
```

这个请求会先进入 Tomcat，再交给 Spring 处理。

### 0.2 Tomcat、Spring Boot、Spring MVC 的关系

这三个东西可以这样理解：

| 技术 | 在项目里负责什么 |
|---|---|
| Tomcat | 接收 HTTP 请求，管理网络连接 |
| Spring Boot | 启动项目，自动配置各种组件 |
| Spring MVC | 把 URL 请求分发到对应 Controller 方法 |

请求进入后大概是：

```text
Tomcat 收到请求
  -> 交给 Spring MVC 的 DispatcherServlet
  -> DispatcherServlet 根据 @RequestMapping/@GetMapping 找 Controller
  -> 调用对应 Java 方法
```

`DispatcherServlet` 可以理解成 Spring MVC 的总调度员。它负责判断：

```text
这个请求 URL 是什么？
HTTP 方法是 GET 还是 POST？
应该调用哪个 Controller 的哪个方法？
请求参数怎么转成 Java 对象？
返回对象怎么转成 JSON？
```

例如本项目：

```java
@RestController
@RequestMapping("/api/grades")
public class GradeController {
    @GetMapping
    public ApiResponse<List<Grade>> list() {
        return ApiResponse.ok(gradeService.visibleGrades());
    }
}
```

当请求是：

```text
GET /api/grades
```

Spring MVC 就会匹配到：

```text
GradeController.list()
```

### 0.3 为什么 Java 对象能自动变成 JSON

因为项目引入了：

```xml
spring-boot-starter-web
```

它默认集成 Jackson。Jackson 是 Java 中常用的 JSON 序列化工具。

后端方法返回：

```java
ApiResponse.ok(data)
```

这是一个 Java 对象，但真正返回给前端时，会被 Jackson 转成 JSON：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

反过来，前端 POST 过来的 JSON：

```json
{
  "studentId": 1,
  "courseId": 2,
  "usualScore": 85,
  "finalScore": 90
}
```

Controller 里写：

```java
public ApiResponse<Grade> save(@RequestBody GradeSaveRequest request)
```

Spring MVC 也会用 Jackson 把 JSON 转成 `GradeSaveRequest` 对象。

### 0.4 为什么很多类不用 new

本项目里很少看到：

```java
new GradeService()
new CourseMapper()
```

原因是 Spring 容器会帮我们管理对象。被这些注解标记的类，会成为 Spring Bean：

```java
@RestController
@Service
@Component
@Configuration
```

例如：

```java
@Service
public class GradeService {
    private final GradeMapper gradeMapper;
    private final CourseService courseService;

    public GradeService(GradeMapper gradeMapper, CourseService courseService) {
        this.gradeMapper = gradeMapper;
        this.courseService = courseService;
    }
}
```

这叫构造器注入。Spring 启动时会：

```text
创建 GradeService 对象
找到它需要 GradeMapper 和 CourseService
自动把对应 Bean 传进构造方法
```

这就是依赖注入。好处是：

1. 类之间不需要自己创建依赖。
2. 对象生命周期由 Spring 管理。
3. 代码结构更清晰。
4. 测试时更容易替换依赖。

### 0.5 为什么 Mapper 接口没有实现类也能用

本项目的 Mapper 只是接口：

```java
public interface GradeMapper {
    List<Grade> findAll();
    int insert(Grade item);
}
```

没有写：

```java
public class GradeMapperImpl implements GradeMapper
```

但是它仍然能运行，是因为 MyBatis 会在程序启动时为 Mapper 接口创建代理对象。

简单理解：

```text
Service 调用 gradeMapper.findAll()
  -> 实际调用的是 MyBatis 创建的代理对象
  -> 代理对象根据方法名 findAll 找 XML 里的 <select id="findAll">
  -> 执行 XML 中的 SQL
  -> 把结果封装成 Java 对象返回
```

这就是 MyBatis Mapper 接口 + XML 的运行原理。

### 0.6 为什么后端必须做权限判断

前端页面可以隐藏按钮，但隐藏按钮不等于安全。

比如学生页面没有“录入成绩”按钮，但如果有人直接用工具请求：

```text
POST /api/grades
```

如果后端不判断权限，就会出问题。

所以本项目真正的安全逻辑都在后端：

```java
if (SecurityUtils.isStudent()) {
    throw new BusinessException("学生不能录入或修改成绩");
}
```

还有教师课程归属：

```java
if (!course.teacherId.equals(currentTeacherId())) {
    throw new BusinessException("教师只能管理本人负责课程");
}
```

后端安全的原则是：

```text
不能相信前端传来的角色；
不能相信前端传来的用户 ID；
不能相信前端传来的总评和绩点；
必须在后端根据 token、数据库和业务规则重新判断。
```

## 1. 后端整体是什么

本项目后端是一个 Spring Boot 单体应用。所谓“单体应用”，就是登录、用户管理、学生管理、教师管理、课程管理、成绩规则、成绩录入、统计分析这些功能都在同一个 Spring Boot 项目里完成，不拆成多个微服务。

后端目录：

```text
backend/src/main/java/com/example/campusgrade
├─ CampusGradeApplication.java       项目启动入口
├─ common/                           通用响应、业务异常、全局异常
├─ config/                           Spring Security 安全配置
├─ controller/                       接口层，接收 HTTP 请求
├─ dto/                              接口请求对象、返回对象
├─ entity/                           数据库实体对象
├─ mapper/                           MyBatis Mapper 接口
├─ security/                         JWT、当前登录用户、权限工具
└─ service/                          业务逻辑层

backend/src/main/resources
├─ application.yml                   后端配置文件
└─ mapper/*.xml                      MyBatis SQL 文件
```

可以用一句话概括后端：

```text
Spring Boot 负责启动和提供 Web 接口；
Spring Security + JWT 负责判断用户是谁；
Controller 负责接收请求；
Service 负责业务和权限；
Mapper + XML 负责访问 MySQL；
ApiResponse 负责统一返回结果。
```

## 2. 后端一次完整请求的技术流程

先不要急着看具体业务，先把所有接口都共用的流程记住。

```text
浏览器 Vue 页面
  -> Axios 发送 HTTP 请求
  -> Spring Boot 内置 Tomcat 接收请求
  -> Spring Security 过滤器链
  -> JwtAuthenticationFilter 解析 token
  -> 找到 Controller 方法
  -> Controller 调用 Service
  -> Service 做业务校验、权限判断、成绩计算
  -> Service 调用 Mapper 接口
  -> MyBatis 根据 Mapper 方法找到 XML SQL
  -> MySQL 执行 SQL
  -> Mapper 返回 Java 对象
  -> Service 返回业务结果
  -> Controller 包装 ApiResponse
  -> 前端收到 JSON 并显示
```

这个流程里最重要的是三层：

| 层 | 本项目位置 | 主要职责 |
|---|---|---|
| Controller | `controller/*.java` | 接收请求、读取参数、返回统一结果 |
| Service | `service/*.java` | 写业务逻辑、权限判断、成绩计算 |
| Mapper | `mapper/*.java` + `resources/mapper/*.xml` | 执行数据库 SQL |

答辩时讲后端流程，可以直接说：

后端采用 Controller、Service、Mapper 三层结构。请求先进入 Controller，Controller 不直接写复杂业务，而是调用 Service；Service 负责业务规则，例如成绩计算和教师课程归属校验；Service 再调用 Mapper，Mapper 通过 MyBatis XML 执行 SQL 操作 MySQL。最后 Controller 用统一响应对象 `ApiResponse` 返回给前端。

## 6. 统一返回：ApiResponse

文件：

```text
backend/src/main/java/com/example/campusgrade/common/ApiResponse.java
```

代码结构：

```java
public class ApiResponse<T> {
    private int code;
    private String message;
    private T data;

    public static <T> ApiResponse<T> ok(T data) {
        return new ApiResponse<>(200, "操作成功", data);
    }

    public static <T> ApiResponse<T> fail(String message) {
        return new ApiResponse<>(500, message, null);
    }
}
```

### 6.1 为什么要统一返回

如果每个接口返回格式都不同，前端处理会很麻烦。

统一后，所有接口都类似：

成功：

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {}
}
```

失败：

```json
{
  "code": 500,
  "message": "错误原因",
  "data": null
}
```

前端 Axios 响应拦截器就可以统一判断 `code`。

## 7. 异常处理：业务错误怎么返回给前端

本项目自定义了业务异常：

```text
backend/src/main/java/com/example/campusgrade/common/BusinessException.java
```

业务代码中遇到不符合规则的情况，就抛出：

```java
throw new BusinessException("教师只能管理本人负责课程");
```

全局异常处理器：

```text
backend/src/main/java/com/example/campusgrade/common/GlobalExceptionHandler.java
```

核心代码：

```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(BusinessException.class)
    @ResponseStatus(HttpStatus.BAD_REQUEST)
    public ApiResponse<Void> handleBusiness(BusinessException ex) {
        return ApiResponse.fail(ex.getMessage());
    }
}
```

### 7.1 `@RestControllerAdvice` 是什么

它是 Spring MVC 的全局异常处理机制。意思是：

```text
Controller 或 Service 中抛出的异常，不需要每个接口 try-catch；
统一交给 GlobalExceptionHandler 转成 JSON 返回给前端。
```

这样业务代码可以直接写：

```java
if (不满足规则) {
    throw new BusinessException("错误提示");
}
```

而不是每个接口都写重复的异常处理。

## 8. JWT 基础知识

本项目使用 JWT 做登录认证。这里先讲清楚 JWT 是什么。

### 8.1 什么是 JWT

JWT 全称是 JSON Web Token。它是一种登录凭证。

传统登录方式经常用 Session：

```text
用户登录成功
  -> 后端在服务器保存 Session
  -> 浏览器保存 SessionId
  -> 后续请求带 SessionId
  -> 后端根据 SessionId 找用户
```

JWT 方式：

```text
用户登录成功
  -> 后端生成一个 token 返回给前端
  -> 前端保存 token
  -> 后续请求带 token
  -> 后端验证 token 签名并解析用户信息
```

### 8.2 JWT 通常长什么样

JWT 是一长串字符串，通常由三段组成：

```text
header.payload.signature
```

三段含义：

| 部分 | 含义 |
|---|---|
| Header | 说明使用什么算法 |
| Payload | 保存用户信息和过期时间等声明 |
| Signature | 签名，防止 token 被篡改 |

例如 payload 中可以保存：

```text
用户名 username
用户 ID uid
角色 role
签发时间 issuedAt
过期时间 expiration
```

### 8.3 JWT 为什么适合前后端分离

因为前后端分离时，前端和后端通常通过接口通信。JWT 不依赖服务器 Session，后端只要验证 token 就能知道是谁。

优点：

1. 无状态，后端不用保存登录会话。
2. 前端通过请求头携带，适合 Axios 调用接口。
3. 可以放用户 ID、角色等信息。

注意：

JWT 不能随便信任前端自己传的内容，必须验证签名。否则用户可以伪造角色。

## 9. 本项目 JWT 生成流程

文件：

```text
backend/src/main/java/com/example/campusgrade/security/JwtTokenProvider.java
```

核心代码：

```java
public String createToken(User user) {
    Instant now = Instant.now();
    return Jwts.builder()
            .subject(user.username)
            .claim("uid", user.id)
            .claim("role", user.role)
            .issuedAt(Date.from(now))
            .expiration(Date.from(now.plus(expirationHours, ChronoUnit.HOURS)))
            .signWith(secretKey)
            .compact();
}
```

这段代码做了什么：

1. `subject(user.username)`：把用户名作为 token 主体。
2. `claim("uid", user.id)`：额外保存用户 ID。
3. `claim("role", user.role)`：额外保存角色。
4. `issuedAt`：记录签发时间。
5. `expiration`：设置过期时间。
6. `signWith(secretKey)`：用密钥签名。
7. `compact()`：生成最终 token 字符串。

### 9.1 密钥从哪里来

```java
public JwtTokenProvider(@Value("${app.jwt.secret}") String secret,
                        @Value("${app.jwt.expiration-hours}") long expirationHours) {
    this.secretKey = Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8));
    this.expirationHours = expirationHours;
}
```

`@Value("${app.jwt.secret}")` 会读取 `application.yml`：

```yaml
app:
  jwt:
    secret: campus-grade-management-demo-secret-key-2026
    expiration-hours: 12
```

也就是说，本项目 token 默认 12 小时过期。

### 9.2 JWT 解析

```java
public String getUsername(String token) {
    Claims claims = Jwts.parser()
            .verifyWith(secretKey)
            .build()
            .parseSignedClaims(token)
            .getPayload();
    return claims.getSubject();
}
```

这段代码做两件事：

1. 使用同一个密钥验证 token 签名。
2. 签名合法后，从 payload 里取出用户名。

如果 token 被篡改，签名验证会失败。

## 9A. JWT 在本项目里的底层运行原理

这一节专门把 JWT 讲透，因为它是本项目登录认证的核心。

### 9A.1 登录认证到底要解决什么问题

HTTP 协议本身是无状态的。

无状态的意思是：

```text
第一次请求登录成功
第二次请求查询成绩
第三次请求修改规则
```

从 HTTP 协议本身看，这三次请求是互相独立的。服务器不会天然记得“第二次请求的人就是刚才登录成功的人”。

所以登录系统必须解决一个问题：

```text
用户登录成功以后，后续请求怎么证明自己还是同一个用户？
```

JWT 就是用来解决这个问题的。

### 9A.2 传统 Session 和 JWT 的区别

Session 方式：

```text
用户登录成功
  -> 服务器保存一份 Session 数据
  -> 浏览器保存 SessionId
  -> 后续请求带 SessionId
  -> 服务器根据 SessionId 去内存或 Redis 找用户
```

JWT 方式：

```text
用户登录成功
  -> 服务器生成一个带签名的 token
  -> 前端保存 token
  -> 后续请求带 token
  -> 服务器验证 token 签名并解析用户
```

主要区别：

| 对比点 | Session | JWT |
|---|---|---|
| 登录状态保存在哪里 | 服务端 | 前端保存 token，服务端验证 |
| 后端是否需要保存会话 | 需要 | 不需要 |
| 适合场景 | 传统服务端页面 | 前后端分离接口 |
| 本项目采用 | 否 | 是 |

本项目是 Vue + Spring Boot 前后端分离，所以使用 JWT 更自然。

### 9A.3 JWT 为什么不能随便伪造

JWT 不是简单把用户信息 Base64 编码一下。它还有签名。

JWT 三段：

```text
Header.Payload.Signature
```

假设 Payload 里有：

```json
{
  "sub": "teacher01",
  "uid": 2,
  "role": "TEACHER",
  "exp": 过期时间
}
```

如果用户自己把 `"role": "TEACHER"` 改成 `"role": "ADMIN"`，Payload 内容变了，签名就对不上。

后端解析时会执行：

```java
Jwts.parser()
    .verifyWith(secretKey)
    .build()
    .parseSignedClaims(token)
```

`verifyWith(secretKey)` 会用服务端密钥验证签名。密钥只有后端知道，前端不知道，所以前端无法生成合法签名。

结论：

```text
JWT 可以被前端保存；
JWT 的内容不应该保存敏感信息；
JWT 的签名可以防止内容被篡改；
后端必须验证签名后才相信 token。
```

### 9A.4 本项目为什么解析 token 后还查数据库

理论上，JWT 里已经有用户名和角色。本项目解析 token 后仍然执行：

```java
User user = userMapper.findByUsername(username);
```

这样做更安全。

原因：

1. 用户可能已经被管理员停用。
2. 用户角色可能已经被修改。
3. 用户可能已经被删除。
4. 数据库里的用户状态比旧 token 更新。

所以项目不是只看 token，而是：

```text
先验证 token 合法
再根据 token 里的用户名查数据库
再确认用户存在且 status = 1
最后才认为用户已登录
```

### 9A.5 JWT 过滤器为什么能在 Controller 前执行

Spring Security 的核心是过滤器链。过滤器链运行在 Controller 之前。

请求进入后端时，并不是马上进入 Controller，而是先经过一串 Filter：

```text
HTTP 请求
  -> Security Filter Chain
  -> JwtAuthenticationFilter
  -> 其他安全过滤器
  -> DispatcherServlet
  -> Controller
```

本项目通过这行配置把 JWT 过滤器加入链路：

```java
.addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class)
```

意思是：

```text
在 Spring Security 默认用户名密码认证过滤器之前，
先执行我们自己的 jwtAuthenticationFilter。
```

所以 Controller 执行前，当前用户已经被解析并保存到 `SecurityContextHolder` 里。

### 9A.6 `SecurityContextHolder` 为什么重要

JWT 过滤器解析出用户后，会创建：

```java
CurrentUser principal = new CurrentUser(user.id, user.username, user.realName, user.role);
```

然后放进认证对象：

```java
UsernamePasswordAuthenticationToken authentication =
        new UsernamePasswordAuthenticationToken(
                principal,
                null,
                List.of(new SimpleGrantedAuthority("ROLE_" + user.role))
        );
```

最后放入：

```java
SecurityContextHolder.getContext().setAuthentication(authentication);
```

这一步相当于告诉 Spring Security：

```text
当前这个请求已经认证通过；
当前用户是 principal；
当前用户拥有 ROLE_ADMIN / ROLE_TEACHER / ROLE_STUDENT 权限。
```

后面代码才能这样判断：

```java
SecurityUtils.isTeacher()
SecurityUtils.currentUser()
@PreAuthorize("hasRole('ADMIN')")
```

### 9A.7 JWT 登录流程的技术本质

本项目 JWT 登录的技术本质是：

```text
用户名密码只在登录时使用一次；
登录成功后，后端签发一个带签名和过期时间的 token；
前端保存 token；
后续请求带 token；
后端过滤器验证 token；
验证成功后把用户信息放入 Spring Security 上下文；
业务代码从上下文获取当前用户并判断权限。
```

## 10. Spring Security 配置流程

文件：

```text
backend/src/main/java/com/example/campusgrade/config/SecurityConfig.java
```

核心代码：

```java
@Configuration
@EnableMethodSecurity
public class SecurityConfig {
    @Bean
    public SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
        return http
                .csrf(csrf -> csrf.disable())
                .cors(cors -> cors.configurationSource(corsConfigurationSource()))
                .sessionManagement(session -> session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
                .authorizeHttpRequests(auth -> auth
                        .requestMatchers("/api/auth/login").permitAll()
                        .anyRequest().authenticated())
                .addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class)
                .build();
    }
}
```

逐行解释：

### 10.1 `csrf.disable()`

关闭 CSRF 防护。

本项目是前后端分离，接口主要靠 JWT 认证，不使用传统表单 + Session，所以这里关闭 CSRF。

### 10.2 `cors(...)`

配置跨域。

前端开发地址是：

```text
http://localhost:5173
http://127.0.0.1:5173
```

后端地址是：

```text
http://localhost:8080
```

端口不同就属于跨域。后端允许这些前端地址访问接口。

### 10.3 `SessionCreationPolicy.STATELESS`

表示后端不创建 Session。

因为本项目使用 JWT，每次请求都带 token，后端不需要保存登录状态。

### 10.4 登录接口放行

```java
.requestMatchers("/api/auth/login").permitAll()
```

登录接口必须允许未登录用户访问，否则用户没法登录。

### 10.5 其他接口必须登录

```java
.anyRequest().authenticated()
```

除登录接口外，其他接口都必须带合法 token。

### 10.6 加入 JWT 过滤器

```java
.addFilterBefore(jwtAuthenticationFilter, UsernamePasswordAuthenticationFilter.class)
```

意思是：

在 Spring Security 默认用户名密码过滤器之前，先执行我们自己的 JWT 过滤器。这样请求进入 Controller 之前，系统就已经知道当前用户是谁。

## 11. JWT 过滤器：每次请求怎么识别当前用户

文件：

```text
backend/src/main/java/com/example/campusgrade/security/JwtAuthenticationFilter.java
```

核心代码：

```java
String header = request.getHeader("Authorization");
if (header != null && header.startsWith("Bearer ")) {
    try {
        String username = jwtTokenProvider.getUsername(header.substring(7));
        User user = userMapper.findByUsername(username);
        if (user != null && user.status != null && user.status == 1) {
            CurrentUser principal = new CurrentUser(user.id, user.username, user.realName, user.role);
            UsernamePasswordAuthenticationToken authentication = new UsernamePasswordAuthenticationToken(
                    principal, null, List.of(new SimpleGrantedAuthority("ROLE_" + user.role)));
            SecurityContextHolder.getContext().setAuthentication(authentication);
        }
    } catch (Exception ignored) {
        SecurityContextHolder.clearContext();
    }
}
chain.doFilter(request, response);
```

### 11.1 `Authorization: Bearer token`

前端请求头格式：

```text
Authorization: Bearer eyJhbGciOi...
```

`Bearer` 是一种通用写法，表示后面这串 token 是身份凭证。

代码里：

```java
header.substring(7)
```

是为了去掉：

```text
"Bearer "
```

只留下真正的 token。

### 11.2 为什么解析 token 后还要查数据库

代码：

```java
User user = userMapper.findByUsername(username);
```

原因：

1. 确认用户仍然存在。
2. 确认用户状态仍然启用。
3. 获取最新的真实姓名、角色等信息。

如果用户已经被管理员停用，即使之前拿到过 token，也不能继续访问。

### 11.3 `SecurityContextHolder` 是什么

```java
SecurityContextHolder.getContext().setAuthentication(authentication);
```

Spring Security 会把当前请求的登录用户放到 `SecurityContextHolder` 中。后面代码就可以通过它获取当前用户。

本项目封装了工具类：

```text
backend/src/main/java/com/example/campusgrade/security/SecurityUtils.java
```

例如：

```java
public static CurrentUser currentUser() {
    Authentication authentication = SecurityContextHolder.getContext().getAuthentication();
    if (authentication == null || !(authentication.getPrincipal() instanceof CurrentUser user)) {
        throw new BusinessException("未登录或登录已过期");
    }
    return user;
}
```

### 12.3 AuthService 登录逻辑

文件：

```text
backend/src/main/java/com/example/campusgrade/service/AuthService.java
```

流程：

```text
1. 根据用户名查 users 表
2. 判断用户是否存在
3. 判断用户状态是否启用
4. 校验密码
5. 清空密码字段
6. 生成 JWT
7. 返回 token 和用户信息
```

核心代码：

```java
User user = userMapper.findByUsername(request.username);
if (user == null || user.status == null || user.status != 1) {
    throw new BusinessException("账号不存在或已停用");
}
if (!passwordMatches(request.password, user.password)) {
    throw new BusinessException("用户名或密码错误");
}
user.password = null;
return new LoginResponse(jwtTokenProvider.createToken(user), user);
```

### 12.4 密码校验

```java
private boolean passwordMatches(String rawPassword, String storedPassword) {
    if (storedPassword != null && storedPassword.startsWith("$2")) {
        return passwordEncoder.matches(rawPassword, storedPassword);
    }
    return rawPassword != null && rawPassword.equals(storedPassword);
}
```

本项目同时兼容两种密码：

| 密码类型 | 判断方式 |
|---|---|
| BCrypt 加密密码 | 用 `passwordEncoder.matches` |
| 演示数据明文密码 | 直接字符串比较 |

正式项目更推荐全部使用 BCrypt 加密保存。

## 13. 角色和权限技术流程

本项目有三类角色：

```text
ADMIN    管理员
TEACHER  教师
STUDENT  学生
```

角色保存在 `users.role` 字段中。

### 13.1 接口级权限：`@PreAuthorize`

示例：

```java
@RestController
@RequestMapping("/api/users")
@PreAuthorize("hasRole('ADMIN')")
public class UserController {
    ...
}
```

含义：

只有拥有 `ROLE_ADMIN` 权限的用户才能访问这个 Controller。

注意：

代码里写的是：

```java
hasRole('ADMIN')
```

Spring Security 实际判断的是：

```text
ROLE_ADMIN
```

而 JWT 过滤器里正好设置了：

```java
new SimpleGrantedAuthority("ROLE_" + user.role)
```

如果数据库角色是 `ADMIN`，最终权限就是 `ROLE_ADMIN`。

### 13.2 业务级权限：教师只能操作自己的课程

只判断角色不够。

例如教师 A 和教师 B 都是 `TEACHER`，但是教师 A 不能修改教师 B 的课程成绩。

所以项目在 `CourseService` 中写了业务归属校验：

```java
public void assertTeacherOwnsCourse(Long courseId) {
    if (!SecurityUtils.isTeacher()) {
        return;
    }
    Course course = findRequiredCourse(courseId);
    if (!course.teacherId.equals(currentTeacherId())) {
        throw new BusinessException("教师只能管理本人负责课程");
    }
}
```

这段代码的逻辑：

1. 如果不是教师，例如管理员，直接放行。
2. 如果是教师，查询课程。
3. 拿课程的 `teacherId` 和当前登录教师 ID 比较。
4. 不一致就抛异常。

这就是业务权限。

接口级权限回答“你是什么角色”，业务级权限回答“这条数据是不是你的”。

## 14. MyBatis 技术流程

本项目持久层使用原生 MyBatis：Mapper 接口 + XML SQL。

### 14.1 Mapper 接口

文件：

```text
backend/src/main/java/com/example/campusgrade/mapper/GradeMapper.java
```

代码：

```java
public interface GradeMapper {
    List<Grade> findAll();
    List<Grade> findByTeacherId(Long teacherId);
    List<Grade> findByStudentId(Long studentId);
    Grade findById(Long id);
    int insert(Grade item);
    int update(Grade item);
    int delete(Long id);
}
```

接口里没有实现类。MyBatis 启动后会自动创建代理对象。

### 14.2 XML SQL

文件：

```text
backend/src/main/resources/mapper/GradeMapper.xml
```

开头：

```xml
<mapper namespace="com.example.campusgrade.mapper.GradeMapper">
```

这个 `namespace` 必须对应 Mapper 接口全路径。

XML 中的方法 ID：

```xml
<select id="findByStudentId" resultType="com.example.campusgrade.entity.Grade">
    <include refid="gradeSelect"/> WHERE g.student_id=#{studentId} ORDER BY c.course_name
</select>
```

对应 Java 接口：

```java
List<Grade> findByStudentId(Long studentId);
```

### 14.3 参数怎么传到 SQL

SQL 里：

```xml
WHERE g.student_id=#{studentId}
```

`#{studentId}` 会取 Java 方法参数：

```java
findByStudentId(Long studentId)
```

MyBatis 会使用预编译参数，不是简单字符串拼接，所以可以减少 SQL 注入风险。

### 14.4 查询结果怎么变成 Java 对象

```xml
resultType="com.example.campusgrade.entity.Grade"
```

表示查询结果会封装成 `Grade` 对象。

数据库字段：

```text
student_no
course_name
total_score
```

Java 字段：

```java
public String studentNo;
public String courseName;
public BigDecimal totalScore;
```

因为配置了：

```yaml
map-underscore-to-camel-case: true
```

所以可以自动映射。

### 14.5 MyBatis 执行 SQL 的底层流程

以：

```java
gradeMapper.findByStudentId(studentId)
```

为例，真正运行时大概是：

```text
1. Spring 注入的 gradeMapper 不是普通对象，而是 MyBatis 代理对象。

2. 调用 findByStudentId 方法时，代理对象会拿到：
   - Mapper 接口全名：com.example.campusgrade.mapper.GradeMapper
   - 方法名：findByStudentId

3. MyBatis 根据 namespace + id 找 XML：
   namespace = com.example.campusgrade.mapper.GradeMapper
   id = findByStudentId

4. 找到 SQL：
   SELECT ... WHERE g.student_id=#{studentId}

5. MyBatis 把 Java 参数 studentId 绑定到 SQL 参数。

6. MyBatis 通过 JDBC 把 SQL 发给 MySQL。

7. MySQL 返回 ResultSet。

8. MyBatis 把 ResultSet 每一行转换成 Grade 对象。

9. 最后返回 List<Grade>。
```

这里的关键是：

```text
Mapper 接口负责 Java 方法；
XML 负责 SQL；
MyBatis 负责把两者连接起来。
```

### 14.6 `#{}` 和 SQL 注入

XML 中使用：

```xml
WHERE g.student_id=#{studentId}
```

`#{}` 不是简单字符串拼接，而是预编译参数绑定，类似 JDBC 的 `PreparedStatement`。

大概等价于：

```sql
WHERE g.student_id = ?
```

再把 `studentId` 作为参数传进去。

这样比直接拼接字符串安全。

不推荐写：

```xml
WHERE g.student_id=${studentId}
```

`${}` 是字符串替换，容易有 SQL 注入风险。本项目主要使用 `#{}`。

### 14.7 MyBatis 和 JDBC 的关系

JDBC 是 Java 访问数据库的底层 API。MyBatis 不是替代数据库驱动，而是封装 JDBC。

底层关系：

```text
Service
  -> Mapper
  -> MyBatis
  -> JDBC
  -> MySQL Connector/J
  -> MySQL
```

也就是说：

```text
MyBatis 帮我们少写大量 JDBC 模板代码；
真正连接 MySQL 仍然依赖 mysql-connector-j 驱动。
```

### 15.1 Entity 是什么

Entity 对应数据库表或查询结果。

例如：

代码：

后半部分是列表查询时 JOIN 出来的展示字段：

```text
student_no, student_name, course_name, teacher_name
```

### 15.2 DTO 是什么

DTO 是 Data Transfer Object，也就是接口传输对象。

它不一定完全对应数据库表。

例如录入成绩请求：

```text
backend/src/main/java/com/example/campusgrade/dto/GradeSaveRequest.java
```

```java
public class GradeSaveRequest {

    @NotNull(message = "课程不能为空")
    public Long courseId;
    @NotNull(message = "平时成绩不能为空")

    @NotNull(message = "期末成绩不能为空")

    public BigDecimal totalScore;
    public String remark;
```

这里虽然有 `totalScore` 字段，但保存时后端不使用前端传来的总评，而是重新计算。

答辩时可以说：

DTO 用来描述接口输入输出，Entity 用来描述数据库数据。把它们分开，可以让接口更清晰，也避免直接把数据库结构完全暴露给前端。

## 18. 课程级成绩规则业务流程

这是后端最核心的业务之一。

### 18.1 为什么要课程级规则

如果所有课程共用一套规则，会有问题。

例如：

```text
高等数学：平时 30%，期末 70%
Java 程序设计：平时 40%，期末 60%
```

如果系统只支持全局规则，就不能体现不同课程考核方式不同。

所以本项目设计为：

```text
courses 表保存每门课程的平时/期末权重
course_grade_rules 表保存每门课程自己的绩点换算区间
```

### 18.2 数据表设计

`courses` 表关键字段：

```sql
usual_weight DECIMAL(5,2) NOT NULL DEFAULT 30.00,
final_weight DECIMAL(5,2) NOT NULL DEFAULT 70.00
```

`course_grade_rules` 表：

```sql
CREATE TABLE course_grade_rules (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  course_id BIGINT NOT NULL,
  min_score DECIMAL(5,2) NOT NULL,
  max_score DECIMAL(5,2) NOT NULL,
  grade_point DECIMAL(3,1) NOT NULL,
  label VARCHAR(50) NOT NULL
);
```

一门课程对应多条绩点规则。

### 18.3 获取规则流程

### 18.4 保存规则流程

接口：

```text

```

Service：

```java

        throw new BusinessException("课程成绩规则不能为空");
    }
    gradeRuleService.validateWeights(config.usualWeight, config.finalWeight);
    courseService.updateWeights(courseId, config.usualWeight, config.finalWeight);
    saveRules(courseId, config.rules);

```

这段代码要重点理解：

2. `updateWeights`：更新课程表中的平时/期末权重。
3. `saveRules`：保存绩点规则。
4. `@Transactional`：保证这些操作要么全部成功，要么全部失败。

### 18.5 保存规则为什么先删后插

代码：

```java

rules.forEach(ruleMapper::insert);
```

因为前端提交的是这门课程完整的一组规则。后端可以先删除旧规则，再插入新规则。这样逻辑简单，避免逐条判断新增、修改、删除。

因为方法加了事务，如果插入中间失败，删除也会回滚。

### 19.1 权重校验

```java
public void validateWeights(BigDecimal usualWeight, BigDecimal finalWeight) {
    if (usualWeight == null || finalWeight == null) {
        throw new BusinessException("平时成绩占比和期末成绩占比不能为空");
    }
    if (usualWeight.compareTo(MIN_SCORE) < 0 || usualWeight.compareTo(MAX_SCORE) > 0
            || finalWeight.compareTo(MIN_SCORE) < 0 || finalWeight.compareTo(MAX_SCORE) > 0) {
        throw new BusinessException("成绩占比必须在 0-100 之间");
    }
    if (usualWeight.add(finalWeight).compareTo(ONE_HUNDRED) != 0) {
        throw new BusinessException("平时成绩占比和期末成绩占比之和必须等于 100");
    }
}
```

逻辑：

1. 平时权重和期末权重不能为空。
2. 两个权重都必须在 0 到 100 之间。
3. 两个权重相加必须等于 100。

为什么使用 `compareTo`：

`BigDecimal` 比较大小不要用 `==`，应该用 `compareTo`。

```java
usualWeight.add(finalWeight).compareTo(ONE_HUNDRED) != 0
```

表示相加结果不等于 100。

### 19.2 分数规则校验

```java
public void validateRules(List<CourseGradeRule> rules) {
    if (rules == null || rules.isEmpty()) {
        throw new BusinessException("绩点规则不能为空");
    }
    List<CourseGradeRule> sortedRules = rules.stream()
            .sorted(Comparator.comparing(rule -> rule.minScore))
            .toList();
    ...
}
```

先检查规则不能为空，再按最低分排序。

为什么排序：

用户在前端输入规则时不一定按顺序，所以后端要自己排序后再校验。

检查第一条：

```java
CourseGradeRule first = sortedRules.get(0);
if (first.minScore.compareTo(MIN_SCORE) != 0) {
    throw new BusinessException("绩点规则必须从 0 分开始");
}
```

说明规则必须从 0 分开始。

检查中间区间：

```java
if (rule.minScore.compareTo(previous.maxScore) <= 0) {
    throw new BusinessException("绩点规则区间不能重叠");
}
BigDecimal gap = rule.minScore.subtract(previous.maxScore);
if (gap.compareTo(MAX_CONTIGUOUS_STEP) > 0) {
    throw new BusinessException("绩点规则必须覆盖 0-100 分");
}
```

含义：

1. 当前最低分不能小于等于上一条最高分，否则重叠。
2. 当前最低分和上一条最高分之间不能有明显空档，否则有缺口。

检查最后一条：

```java
CourseGradeRule last = sortedRules.get(sortedRules.size() - 1);
if (last.maxScore.compareTo(MAX_SCORE) != 0) {
    throw new BusinessException("绩点规则必须覆盖到 100 分");
}
```

说明规则必须覆盖到 100 分。

### 19.3 单条规则校验

```java
private void validateSingleRule(CourseGradeRule rule) {
    if (rule.minScore == null || rule.maxScore == null || rule.gradePoint == null) {
        throw new BusinessException("绩点规则分数和绩点不能为空");
    }
    if (rule.minScore.compareTo(MIN_SCORE) < 0 || rule.maxScore.compareTo(MAX_SCORE) > 0) {
        throw new BusinessException("绩点规则区间必须在 0-100 分之间");
    }
    if (rule.minScore.compareTo(rule.maxScore) >= 0) {
        throw new BusinessException("绩点规则最小分必须小于最大分");
    }
}
```

这保证每条规则本身合法。

## 21. 总评计算代码讲解

文件：

```text
GradeRuleService.java
```

代码：

```java
public BigDecimal calculateTotalScore(BigDecimal usualScore, BigDecimal finalScore,
                                       BigDecimal usualWeight, BigDecimal finalWeight) {
    validateScore(usualScore, "平时成绩");
    validateScore(finalScore, "期末成绩");
    validateWeights(usualWeight, finalWeight);
    return usualScore.multiply(usualWeight)
            .add(finalScore.multiply(finalWeight))
            .divide(ONE_HUNDRED, 2, RoundingMode.HALF_UP);
}
```

公式：

```text
总评 = 平时分 * 平时权重 / 100 + 期末分 * 期末权重 / 100
```

例如：

```text
平时分 80
期末分 90
平时权重 30
期末权重 70

总评 = 80 * 30 / 100 + 90 * 70 / 100
     = 24 + 63
     = 87.00
```

### 21.1 为什么用 BigDecimal

Java 中 `double` 是二进制浮点数，计算小数时可能出现精度问题。

例如有时会出现：

```text
0.1 + 0.2 = 0.30000000000000004
```

成绩、绩点、权重需要精确小数，所以使用 `BigDecimal`。

### 21.2 为什么保留两位小数

```java
.divide(ONE_HUNDRED, 2, RoundingMode.HALF_UP)
```

含义：

1. 除以 100。
2. 保留 2 位小数。
3. 使用四舍五入。

## 22. 绩点计算代码讲解

代码：

```java
public BigDecimal calculateGradePoint(List<CourseGradeRule> rules, BigDecimal score) {
    validateRules(rules);
    if (score == null || score.compareTo(MIN_SCORE) < 0 || score.compareTo(MAX_SCORE) > 0) {
        throw new BusinessException("成绩必须在 0-100 分之间");
    }
    return rules.stream()
            .filter(rule -> score.compareTo(rule.minScore) >= 0 && score.compareTo(rule.maxScore) <= 0)
            .findFirst()
            .map(rule -> rule.gradePoint)
            .orElseThrow(() -> new BusinessException("未找到匹配的课程绩点规则"));
}
```

流程：

1. 先校验规则合法。
2. 判断总评分数是否在 0 到 100 之间。
3. 遍历这门课程的规则列表。
4. 找到 `minScore <= score <= maxScore` 的区间。
5. 返回该区间的 `gradePoint`。

例如：

```text
高等数学规则：
80 - 89.99 -> 3.7
90 - 100   -> 4.0

总评 85
命中 80 - 89.99
绩点 3.7
```

重点：

这里传入的是当前课程的规则列表，不是全局规则。所以同样 85 分，在不同课程里可以得到不同绩点。

## 26A. 事务 Transaction 原理和本项目用法

事务是数据库里非常重要的概念。它解决的问题是：

```text
多个数据库操作要么全部成功，要么全部失败。
```

### 26A.1 为什么保存成绩规则需要事务

本项目保存课程成绩规则时，不是只操作一次数据库，而是连续操作多次：

```text
1. 更新 courses 表里的 usual_weight、final_weight
2. 删除 course_grade_rules 表中该课程旧规则
3. 插入 course_grade_rules 表中新规则
```

如果没有事务，可能出现这种情况：

```text
课程权重更新成功
旧规则删除成功
新规则插入到一半失败
```

这样数据库就会处于不完整状态。

所以代码使用：

```java
@Transactional
public CourseGradeRuleConfig saveConfig(Long courseId, CourseGradeRuleConfig config) {
    ...
}
```

### 26A.2 `@Transactional` 做了什么

可以理解为 Spring 在方法外面包了一层事务控制：

```text
开启事务
  -> 执行业务方法
  -> 如果全部成功，提交事务 commit
  -> 如果中间抛异常，回滚事务 rollback
```

伪代码：

```java
try {
    beginTransaction();
    saveConfig();
    commit();
} catch (Exception e) {
    rollback();
    throw e;
}
```

本项目中，如果保存规则时校验失败或插入失败，前面已经做过的数据库操作也会回滚。

### 26A.3 事务和业务异常的关系

本项目业务异常是：

```java
BusinessException
```

如果在 `@Transactional` 方法中抛出运行时异常，Spring 会回滚事务。

例如：

```java
gradeRuleService.validateWeights(config.usualWeight, config.finalWeight);
```

如果权重之和不是 100，就抛出 `BusinessException`，后续数据库操作不会继续执行。

### 26A.4 为什么不是所有方法都加事务

查询方法一般不需要事务。

例如：

```java
visibleGrades()
findByStudentId()
overview()
```

只是查数据，没有多步写操作，不加事务也可以。

保存课程规则这种多步写操作更需要事务。

## 26B. 参数校验 Validation 原理

本项目使用了 Spring Boot Validation。

例如：

```java
public class GradeSaveRequest {
    @NotNull(message = "学生不能为空")
    public Long studentId;

    @NotNull(message = "课程不能为空")
    public Long courseId;

    @NotNull(message = "平时成绩不能为空")
    public BigDecimal usualScore;

    @NotNull(message = "期末成绩不能为空")
    public BigDecimal finalScore;
}
```

Controller 中：

```java
public ApiResponse<Grade> save(@Valid @RequestBody GradeSaveRequest request)
```

`@Valid` 的作用是触发 DTO 上的校验注解。

如果前端没传 `usualScore`，Spring 会在进入业务逻辑前发现参数不合法，然后抛出 `MethodArgumentNotValidException`。

全局异常处理器会捕获：

```java
@ExceptionHandler(MethodArgumentNotValidException.class)
@ResponseStatus(HttpStatus.BAD_REQUEST)
public ApiResponse<Void> handleValidation(MethodArgumentNotValidException ex) {
    String message = ex.getBindingResult().getFieldErrors().isEmpty()
            ? "参数校验失败"
            : ex.getBindingResult().getFieldErrors().get(0).getDefaultMessage();
    return ApiResponse.fail(message);
}
```

所以参数校验流程是：

```text
前端提交 JSON
  -> Spring 把 JSON 转 DTO
  -> @Valid 检查 DTO 字段
  -> 不合法则抛校验异常
  -> GlobalExceptionHandler 返回错误 JSON
  -> 前端提示错误
```

## 26C. BigDecimal 原理和本项目用法

成绩计算没有使用 `double`，而是使用 `BigDecimal`。

原因是 `double` 适合科学计算，但不适合精确小数。

例如：

```java
0.1 + 0.2
```

在二进制浮点数中可能不是精确的 `0.3`。

本项目涉及：

```text
平时分
期末分
总评
绩点
学分
权重
```

这些都需要比较稳定的小数计算，所以使用 `BigDecimal`。

### 26C.1 BigDecimal 比较大小

不能用：

```java
usualWeight == finalWeight
```

应该用：

```java
usualWeight.compareTo(finalWeight)
```

`compareTo` 返回：

| 返回值 | 含义 |
|---|---|
| `< 0` | 小于 |
| `= 0` | 等于 |
| `> 0` | 大于 |

例如：

```java
usualWeight.add(finalWeight).compareTo(ONE_HUNDRED) != 0
```

表示两个权重相加不等于 100。

### 26C.2 BigDecimal 计算总评

```java
return usualScore.multiply(usualWeight)
        .add(finalScore.multiply(finalWeight))
        .divide(ONE_HUNDRED, 2, RoundingMode.HALF_UP);
```

分成三步：

```text
usualScore.multiply(usualWeight)
    平时分 * 平时权重

finalScore.multiply(finalWeight)
    期末分 * 期末权重

divide(100, 2, HALF_UP)
    除以 100，保留两位小数，四舍五入
```

例如：

```text
80 * 30 + 90 * 70 = 8700
8700 / 100 = 87.00
```

## 26D. 后端为什么不相信前端传来的计算结果

`GradeSaveRequest` 里虽然有：

```java
public BigDecimal totalScore;
```

但是 `GradeService.save` 保存时没有使用它。

真正保存的是：

```java
BigDecimal totalScore = gradeRuleService.calculateTotalScore(...);
BigDecimal gradePoint = gradeRuleService.calculateGradePoint(...);

grade.totalScore = totalScore;
grade.gradePoint = gradePoint;
```

原因：

```text
前端可以被用户修改；
请求参数可以被工具伪造；
总评和绩点属于业务规则结果；
业务规则必须以后端计算为准。
```

所以本项目后端的设计原则是：

```text
原始成绩可以由用户输入；
最终总评和绩点必须由后端根据数据库规则计算。
```

## 28. 按一次保存成绩按钮，从入口到结束完整串讲

这是最适合答辩讲的完整技术链路。

```text
1. 教师在前端成绩录入页面输入学生 ID、课程 ID、平时分、期末分。

2. 点击保存按钮。

3. 前端 SimpleCrud 组件调用 save 方法。

4. save 方法调用 gradeApi.create 或 gradeApi.update。

5. gradeApi 通过 Axios 请求后端 /api/grades。

6. Axios 请求拦截器从 localStorage 取 token，加入 Authorization 请求头。

7. 请求到达 Spring Boot 后端。

8. Spring Security 过滤器链先处理请求。

9. JwtAuthenticationFilter 从 Authorization 中取出 token。

10. JwtTokenProvider 验证 token 签名并解析用户名。

11. UserMapper 根据用户名查询 users 表。

12. 如果用户存在且启用，后端把 CurrentUser 放进 SecurityContextHolder。

13. 请求进入 GradeController.save。

14. Controller 使用 @RequestBody 把 JSON 转成 GradeSaveRequest。

15. Controller 调用 GradeService.save。

16. GradeService 判断当前用户不是学生。

17. GradeService 调用 CourseService.assertTeacherOwnsCourse 校验教师是否拥有课程。

18. CourseService 查询 courses 表获取课程权重。

19. CourseGradeRuleMapper 查询 course_grade_rules 表获取该课程绩点规则。

20. GradeRuleService.calculateTotalScore 计算总评。

21. GradeRuleService.calculateGradePoint 根据课程规则计算绩点。

22. GradeService 组装 Grade 对象。

23. GradeMapper.insert 或 update 调用 GradeMapper.xml 中的 SQL。

24. MySQL 写入 grades 表。

25. 后端重新查询保存后的成绩记录。

26. Controller 用 ApiResponse.ok 返回结果。

27. 前端 Axios 响应拦截器取出 data。

28. 前端提示保存成功，重新加载成绩表格。
```

这条链路包含了本项目几乎所有核心后端技术：

```text
HTTP 请求
JWT
Spring Security
Controller
DTO
Service
业务校验
BigDecimal 计算
MyBatis Mapper
XML SQL
MySQL
统一返回
```

## 29. 按登录流程，从入口到结束完整串讲

```text
1. 用户在前端登录页输入用户名和密码。

2. 前端调用 authApi.login。

3. Axios 发送 POST /api/auth/login。

4. SecurityConfig 放行 /api/auth/login，因为登录前没有 token。

5. 请求进入 AuthController.login。

6. Spring 用 @RequestBody 把 JSON 转成 LoginRequest。

7. AuthController 调用 AuthService.login。

8. AuthService 调用 UserMapper.findByUsername 查询 users 表。

9. 如果用户不存在或停用，抛 BusinessException。

10. 如果用户存在，校验密码。

11. 密码正确后，JwtTokenProvider.createToken 生成 JWT。

12. 后端把 token 和用户信息封装成 LoginResponse。

13. Controller 返回 ApiResponse.ok。

14. 前端收到 token，保存到 Pinia 和 localStorage。

15. 后续请求 Axios 自动携带 token。
```

登录流程的核心是：

```text
第一次登录用用户名密码；
登录成功后用 JWT；
后续请求不再传用户名密码。
```

## 32. 常见基础技术概念补充

### 32.1 什么是 Controller

Controller 是接口入口。它接收 HTTP 请求。

例如：

```java
@RestController
@RequestMapping("/api/grades")
public class GradeController {
    @GetMapping
    public ApiResponse<List<Grade>> list() {
        return ApiResponse.ok(gradeService.visibleGrades());
    }
}
```

含义：

```text
GET /api/grades 请求会进入 list 方法
```

### 32.2 什么是 Service

Service 是业务层。它负责真正的业务规则。

例如成绩计算、权限判断、规则校验都在 Service。

为什么不写在 Controller？

因为 Controller 应该保持简单，只做请求入口。业务写在 Service，结构更清晰，也方便多个接口复用同一段业务逻辑。

### 32.3 什么是 Mapper

Mapper 是数据库访问层。

Service 不直接写 SQL，而是调用 Mapper 方法：

```java
gradeMapper.findByStudentId(studentId)
```

MyBatis 再去 XML 找对应 SQL。

### 32.4 什么是 Bean

在 Spring 中，被 Spring 容器管理的对象叫 Bean。

例如：

```java
@Service
public class GradeService { ... }
```

Spring 启动后会创建 `GradeService` 对象，并在需要的地方自动注入。

### 32.5 什么是依赖注入

例如：

```java
public GradeService(GradeMapper gradeMapper, StudentMapper studentMapper) {
    this.gradeMapper = gradeMapper;
    this.studentMapper = studentMapper;
}
```

`GradeService` 需要 `GradeMapper` 和 `StudentMapper`，Spring 会自动把这些对象传进构造方法。

这叫依赖注入。

好处：

```text
类之间不用自己 new 对象；
由 Spring 统一创建和管理；
代码更容易维护和测试。
```

### 32.6 什么是 REST API

REST API 是一种接口设计风格。

本项目中：

```text
GET    /api/grades       查询成绩
POST   /api/grades       新增成绩
PUT    /api/grades/{id}  修改成绩
DELETE /api/grades/{id}  删除成绩
```

路径表示资源，HTTP 方法表示操作。

## 34. 最后总结：后端技术主线

这个项目后端的主线可以概括成：

```text
Spring Boot 启动项目
  -> 读取 application.yml 连接 MySQL
  -> Spring Security 建立安全过滤链
  -> 登录接口校验账号密码并签发 JWT
  -> 前端后续请求携带 JWT
  -> JwtAuthenticationFilter 解析当前用户
  -> Controller 接收接口请求
  -> Service 根据角色和业务数据判断权限
  -> Service 完成课程级成绩规则校验和成绩计算
  -> Mapper + XML 执行 SQL 访问 MySQL
  -> ApiResponse 统一返回给前端
```

本项目最能体现 Java 后端设计的地方有三个：

1. **JWT 登录认证**：登录后生成 token，后续请求通过 token 识别用户。
2. **角色权限 + 业务权限**：管理员、教师、学生权限不同，教师还要校验课程归属。
3. **课程级成绩计算规则**：每门课程独立配置权重和绩点规则，后端自动计算总评和绩点。

只要把这三条讲清楚，整个后端技术流程就基本讲清楚了。
