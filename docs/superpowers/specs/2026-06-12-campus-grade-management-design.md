# 高校成绩管理系统设计规格

## 1. 项目目标

本项目实现一个用于课程设计提交与答辩的 Web 全栈演示系统，名称为“高校成绩管理系统”。系统采用 Spring Boot 后端、Vue 3 前端、MySQL 8 数据库，覆盖管理员、教师、学生三类用户的登录、权限、基础数据管理、课程级绩点规则、成绩录入查询和统计展示。

项目定位为“标准完整演示版”：功能完整、可运行、可演示，避免移动端、选课、审批、导入导出、微服务等超出课程设计范围的大功能。

## 2. 技术架构

系统采用前后端分离架构：

- 前端：Vue 3、Vite、Element Plus、Pinia、Axios。
- 后端：Spring Boot、Spring Security、JWT、原生 MyBatis Mapper 接口与 XML SQL。
- 数据库：MySQL 8，默认数据库名为 `campus_grade_management`。
- 文档：中文 Word 课程设计报告，内容与实现的页面、表结构、接口、权限、测试结果保持一致。

后端按 `controller`、`service`、`mapper`、`entity`、`dto`、`config`、`security` 分层。前端按 `api`、`router`、`stores`、`layout`、`views`、`utils` 组织。

## 3. 角色与权限

系统包含三类角色：

- 管理员：管理用户、学生、教师、班级、课程、课程绩点规则、成绩和统计数据。
- 教师：查看自己的课程，只能为自己负责的课程录入和修改成绩，查看自己课程的统计。
- 学生：查看个人信息、本人课程成绩和成绩汇总，不能访问其他学生成绩。

JWT 登录成功后返回 token、用户基本信息和角色。前端根据角色动态生成菜单，后端接口继续做服务端权限校验，避免只依赖前端隐藏菜单。

## 4. 核心功能模块

管理员页面：

- 用户管理：新增、编辑、停用用户，设置角色。
- 学生管理：维护学号、姓名、班级、联系电话、关联登录账号。
- 教师管理：维护工号、姓名、职称、联系电话、关联登录账号。
- 班级管理：维护班级名称、年级、专业。
- 课程管理：维护课程编号、课程名称、学分、授课教师。
- 绩点规则管理：按课程维护分数区间与绩点映射。
- 成绩管理：为任意课程和学生录入成绩，系统自动计算绩点。
- 统计分析：展示课程平均分、最高分、最低分、及格率、成绩分布。

教师页面：

- 我的课程：查看本人负责的课程。
- 成绩录入：只管理本人课程成绩。
- 课程统计：查看本人课程统计。

学生页面：

- 个人信息：查看本人学生档案。
- 我的成绩：查看本人各课程成绩。
- 成绩汇总：查看平均分、总学分、平均绩点和通过课程数。

## 5. 数据库设计

核心表如下：

- `users`：登录账号、密码哈希、姓名、角色、状态。
- `classes`：班级基础信息。
- `students`：学生档案，关联 `users` 和 `classes`。
- `teachers`：教师档案，关联 `users`。
- `courses`：课程信息，关联授课教师。
- `course_grade_rules`：课程级绩点规则，字段为 `course_id`、`min_score`、`max_score`、`grade_point`、`label`。
- `grades`：成绩记录，关联学生和课程，保存平时成绩、期末成绩、总评成绩、绩点。

数据库种子数据包含：

- 管理员账号：`admin / admin123`。
- 教师账号：`teacher01 / 123456`。
- 学生账号：`student01 / 123456`。
- 至少两个课程配置不同的绩点规则，用同样的总评成绩演示不同课程得出不同绩点。

## 6. 课程级绩点规则

绩点计算必须按课程独立配置，不使用全局统一绩点表。

规则要求：

- 每门课程在录入成绩前必须配置有效规则。
- 同一课程规则区间覆盖 `0-100`。
- 同一课程规则区间不能重叠。
- 成绩录入或更新时，后端根据课程 ID 和总评成绩查询对应区间，将结果写入 `grades.grade_point`。

示例演示：

- 高等数学：85 分映射为 3.7。
- Java 程序设计：85 分映射为 3.3。

这样可在答辩时直观说明“绩点规则是课程级，而不是全局级”。

## 7. 后端接口设计

接口统一以 `/api` 开头。

- `POST /api/auth/login`：登录。
- `GET /api/auth/me`：获取当前用户。
- `/api/users`：用户 CRUD，仅管理员。
- `/api/classes`：班级 CRUD，管理员。
- `/api/students`：学生 CRUD；学生仅可查看本人。
- `/api/teachers`：教师 CRUD；教师可查看本人。
- `/api/courses`：课程 CRUD；教师仅查看本人课程。
- `/api/courses/{courseId}/grade-rules`：课程绩点规则查询与保存。
- `/api/grades`：成绩查询、录入、更新、删除，按角色过滤权限。
- `/api/statistics/overview`：管理员概览统计。
- `/api/statistics/courses/{courseId}`：课程统计，管理员或该课程教师可访问。
- `/api/statistics/student-summary`：学生本人汇总。

后端统一返回结构包含 `code`、`message`、`data`，并提供全局异常处理。

## 8. 前端设计

前端采用后台管理系统风格：

- 登录页居中表单，展示系统名称和演示账号提示。
- 主页面左侧深色侧边栏，顶部显示当前用户和退出按钮。
- 内容区使用 Element Plus 表格、表单、弹窗、统计卡片。
- 管理员、教师、学生登录后看到不同菜单。

界面保持清爽、可读、适合课程设计答辩截图。页面不做移动 App，但保证常规浏览器窗口下布局正常。

## 9. 报告设计

最终生成中文 Word 报告，至少包含：

- 需求分析。
- 系统架构。
- 功能模块设计。
- 数据库设计。
- API/接口设计。
- 关键实现细节。
- 测试计划与测试结果。
- 系统截图。
- 总结。

报告中的表名、字段、页面、角色、接口、测试账号、绩点规则示例必须与实际实现一致。

## 10. 验证方案

完成前验证：

- MySQL schema 和 seed SQL 可加载。
- Spring Boot 后端可启动。
- Vue 前端可启动。
- 管理员、教师、学生账号可登录。
- 角色菜单和后端权限符合设计。
- 管理员可配置课程绩点规则。
- 成绩录入时按课程规则计算绩点。
- 学生不能查看其他学生成绩。
- 教师不能编辑非本人课程成绩。
- 核心页面无明显控制台错误。
- Word 报告内容与系统实现一致。

## 11. 已确认决策

用户已选择方案 1：标准完整演示版。系统按本规格实施，不加入移动端、选课、审批、Excel 导入导出、复杂审计日志、微服务等额外功能。
